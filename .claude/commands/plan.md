# /plan - 구현 계획 수립

3개 파일 이상 변경, 아키텍처 변경, API/DB 스키마 변경 시 반드시 이 커맨드를 먼저 실행하라.

## 동작

1. **요구사항 재정의** — 무엇을 만들어야 하는지 명확히
2. **위험 식별** — 잠재적 이슈와 블로커
3. **단계별 계획** — 구현을 페이즈로 분해
4. **사용자 확인 대기** — 반드시 승인 후 진행

## 출력 형식

```
# 구현 계획: [기능명]

## 요구사항 재정의
- [핵심 요구사항 1]
- [핵심 요구사항 2]

## 구현 페이즈

### Phase 1: [제목]
- [구체적 작업 1]
- [구체적 작업 2]

### Phase 2: [제목]
- [구체적 작업]

## 의존성
- [외부 의존성]

## 위험
- HIGH: [위험 1]
- MEDIUM: [위험 2]

## 예상 복잡도: [HIGH/MEDIUM/LOW]

**확인 대기**: 이 계획으로 진행할까요? (yes/no/수정사항)
```

## 중요

- 사용자가 "yes" 또는 "진행"으로 확인할 때까지 **코드 작성 금지**
- 수정 요청 시 계획 조정 후 다시 확인
- 확정된 계획은 Notion에 보고:
  ```python
  report_decision("구현 계획: [기능명]", "아키텍처", "[계획 요약]", "[선택 이유]", alternatives="[대안]", project_name="[프로젝트명]")
  ```

## CONTEXT.md 참조

/plan 실행 시 .planning/ 에 해당 작업의 CONTEXT.md가 있으면 먼저 읽어라.
CONTEXT.md의 "확정 사항"은 반드시 준수한다.
"Claude 재량" 항목은 최적 판단으로 결정하되, PLAN.md에 결정 이유를 기록한다.
CONTEXT.md가 없으면 기존 /plan 로직대로 진행한다.

/plan 출력은 .planning/{작업명}-PLAN.md에도 저장하라.

## Nyquist 검증 (계획 품질 게이트)

계획 작성 완료 후, 제출 전에 아래를 확인하라:

### 필수 체크 (하나라도 없으면 계획 불합격)
1. **테스트 피드백 루프**: "어떻게 검증할 것인가"가 명시되어 있는가?
   - 빌드/lint/테스트 명령어가 구체적으로 포함되어 있어야 함
   - "나중에 테스트" = 불합격

2. **완료 조건 측정 가능성**: 완료 조건이 주관적이지 않은가?
   - "잘 동작함" = 불합격
   - "pytest 전체 통과 + /health 200 응답" = 합격

3. **수정 범위 명시**: 수정할 파일과 수정하지 않을 파일이 구분되어 있는가?

### 불합격 시
계획을 보완하여 재작성한다. 스크립트로 체크 가능:
```bash
./scripts/nyquist-check.sh .planning/{작업명}-PLAN.md
```
