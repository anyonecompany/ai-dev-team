# Discuss — 실행 전 결정 잠금

$ARGUMENTS

## 목적

코드를 쓰기 전에 "무엇을 어떻게 만들 것인가"를 확정한다.
기술적 구현 방법은 /plan의 영역이다.
여기서는 사용자의 선호도와 결정 사항만 잠근다.

## 프로세스

### 1. 프로젝트 컨텍스트 로드
```bash
./scripts/load-project-context.sh {감지된 프로젝트}
```

프로젝트의 CLAUDE.md, codemap, 관련 knowledge를 읽는다.

### 2. 회색 영역(Gray Areas) 식별

작업 내용을 분석하여 결정이 필요한 항목을 식별:

| 작업 유형 | 회색 영역 예시 |
|-----------|---------------|
| UI 기능 | 레이아웃, 밀도, 인터랙션, 빈 상태 처리 |
| API | 응답 포맷, 에러 처리, 페이지네이션 방식 |
| 데이터 | 캐시 전략, 갱신 주기, 정합성 수준 |
| 알림 | 트리거 조건, 빈도, 채널 |

### 3. 질문 → 확정

각 회색 영역에 대해 사용자에게 질문한다.
사용자가 답하면 "확정 사항"으로 잠근다.
사용자가 "알아서 해"라고 하면 "Claude 재량"으로 표시한다.

### 4. CONTEXT.md 생성

파일: .planning/{작업명 kebab-case}-CONTEXT.md

```markdown
# {작업명} — 결정 컨텍스트
> 생성일: {날짜}
> 프로젝트: {프로젝트명}

## 작업 요약
{한 줄 요약}

## 확정 사항 (Locked)
- {결정 1}: {선택한 옵션}
- {결정 2}: {선택한 옵션}

## Claude 재량 (Flexible)
- {결정 3}: Claude가 최적 판단

## 비범위 (Out of Scope)
- {이번 작업에서 하지 않을 것}

## 참고
- 관련 ADR: {있으면}
- 관련 mistakes: {있으면}
- 관련 patterns: {있으면}
```

### 5. 가정 모드 (Assumptions Mode)

사용자가 바쁘거나 빠르게 진행하고 싶으면:
- 코드베이스를 먼저 분석
- "이렇게 하려고 합니다" 형태로 가정을 제시
- 사용자는 수정할 것만 수정
- 나머지는 자동 확정

가정 모드 트리거: "빠르게 해줘", "알아서 해줘", "가정해서 진행해"

## 출력

CONTEXT.md가 생성되면 다음 단계(/plan)에서 이 파일을 참조.
