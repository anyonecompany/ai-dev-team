# 검증 후 완료 선언 (Verification Before Completion)

> 원칙: 증거 없는 완료 선언은 거짓이다. "아마 될 것이다"는 법정에서 "아마 무죄"만큼 무효하다.
> 출처: claude-forge golden-principles #10 적응

## 철의 법칙

```
실행 증거 없이 완료를 선언할 수 없다.
```

이번 메시지에서 검증 명령을 실행하지 않았다면, 통과했다고 주장할 수 없다. 이전 실행은 무효 — 코드가 바뀌었을 수 있다.

## 게이트 함수

성공, 완료, 만족을 주장하기 전에 반드시:

```
1. IDENTIFY  — 이 주장을 증명할 명령은?
2. RUN       — 전체 명령 실행 (새로, 완전히)
3. READ      — 전체 출력 읽기, exit code 확인, 실패 횟수 세기
4. VERIFY    — 출력이 주장을 확인하는가?
               아니오 → 실제 상태를 증거와 함께 보고
               예 → 증거와 함께 주장 진행
5. CLAIM     — 이제서야 결과를 선언

어떤 단계든 건너뛰면 = 검증이 아니라 추측
```

## 검증 체크리스트

| 주장 | 필요한 증거 | 불충분한 증거 |
|------|-----------|-------------|
| 테스트 통과 | 테스트 명령 출력: 0 실패 | 이전 실행, "통과할 것" |
| 린트 클린 | 린터 출력: 0 에러 | 부분 체크, 가정 |
| 빌드 성공 | 빌드 명령: exit 0 | "린트 통과했으니 빌드도" |
| 버그 수정 | 회귀 테스트: 통과 | "코드 바꿨으니 고쳐졌을 것" |
| 회귀 테스트 작동 | Red-Green 사이클 검증 | 실패 확인 없는 단일 통과 |
| 에이전트 완료 | VCS diff로 변경 확인 | 에이전트 자체 "성공" 보고 |
| 요구사항 충족 | 항목별 체크리스트 검증 | "테스트 통과했으니 요구사항도" |

## Red-Green 검증 (TDD)

회귀 테스트와 버그 수정에서 단일 통과는 부족하다:

```
1. 테스트 작성 → 실행 → PASS (테스트 동작 확인)
2. 수정 되돌림 → 실행 → FAIL (테스트가 버그를 잡는지 확인)
3. 수정 복원   → 실행 → PASS (수정이 버그를 해결하는지 확인)
```

2단계에서 실패하지 않으면, 테스트가 실제로 수정을 검증하지 않는 것이다.

## 위험 신호 — 이 생각이 들면 멈춰라

| 생각 | 현실 |
|------|------|
| "아마 될 것이다" | 검증을 실행하라 |
| "확신한다" | 확신은 증거가 아니다 |
| "이번만 넘어가자" | 예외 없음 |
| "린트 통과했으니..." | 린트 ≠ 컴파일러 ≠ 테스트 |
| "에이전트가 성공이라 했다" | 독립적으로 검증하라 |
| "간단한 변경이니 검증 불필요" | 간단한 변경도 깨진다 |
| "아까 이미 실행했다" | 아까 ≠ 지금. 다시 실행 |
| "~할 것이다", "아마", "~인 듯" | 확률적 언어는 증거가 아니다 |
| "좋아!", "완벽!", "끝!" | 검증 전에 만족을 표현하지 마라 |

## 적용 시점

**항상**, 다음 전에:
- 성공이나 완료를 주장하기 전
- 결과에 만족을 표현하기 전
- 커밋이나 PR 생성 전
- 태스크를 완료로 표시하기 전
- 다음 태스크로 넘어가기 전
- 서브 에이전트에 위임할 때

## 결론

검증에 지름길은 없다. 명령을 실행하라. 출력을 읽어라. 그런 다음에만 결과를 선언하라. 비타협적이다.
