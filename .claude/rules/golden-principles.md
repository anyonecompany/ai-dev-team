# Golden Principles (골든 원칙)

> 깨끗하고 유지보수 가능한 코드를 위한 12가지 핵심 원칙
> 출처: claude-forge + ai-dev-team 통합

## 1. 불변성 (Immutability)

변이(mutation)는 버그의 온상이다. 스프레드 연산자로 새 객체를 생성하라. 원본을 수정하지 마라.

## 2. 환경변수로 시크릿 관리

시크릿을 절대 하드코딩하지 마라. `process.env` 또는 `os.environ`만 사용. 미설정 시 즉시 실패(fail-fast).

## 3. 테스트 우선 (TDD)

구현 후 테스트는 "통과만 하는 테스트"를 만든다. RED → GREEN → IMPROVE. 커버리지 80% 이상.

## 4. 결론 먼저, 이유는 나중에

첫 문장에 결론. "왜냐하면..."은 그 다음. 10줄 분석 후 결론 금지.

## 5. 작은 파일, 작은 함수

파일: 800줄 이내. 함수: 50줄 이내. 중첩: 4단계 이내. 초과 시 분리.

## 6. 시스템 경계에서 검증

내부 코드는 신뢰. 사용자 입력과 외부 API 응답은 검증 (pydantic, zod, parameterized queries).

## 7. 비유로 설명

일상적 비유 1-2문장 먼저, 기술적 설명은 그 뒤.

## 8. 컨텍스트 50% 규칙

컨텍스트 윈도우의 50% 내에 작업 완료. 큰 태스크는 새 세션으로 분리.

## 9. 하드게이트: 설계 없이 코딩 금지

3개 파일 이상 변경, 아키텍처 변경, API 엔드포인트 변경, DB 스키마 변경 시 → 반드시 설계 먼저. 사용자 승인 후 코딩. 예외: 1-2개 파일의 단순 수정/버그 패치.

## 10. 증거 기반 완료

"됐다"는 증거 없이 거짓. 완료 전: 테스트 결과(통과/실패 수, 커버리지), 빌드 성공(실행 확인), 요구사항 체크리스트 검증. 금지: "될 것이다", "문제없을 것". 필수: "12개 테스트 통과", "빌드 성공 (0 에러)".

## 11. 외과적 변경 (Surgical Changes)

요청받은 것만 변경하라. 모든 변경 줄은 사용자 요청에 직접 연결되어야 한다.
- 인접 코드, 주석, 포맷팅 "개선" 금지
- 기존 스타일에 맞춰라, 네가 다르게 할지라도
- 관련 없는 죽은 코드: 언급은 하되 삭제는 하지 마라
- 네 변경이 만든 고아(unused imports 등)만 정리

## 12. 가정 먼저 확인

모호한 요구사항에서 조용히 추측하지 마라. 가정을 명시하고 확인받아라. 더 단순한 접근이 있으면 제안하라.

## 반박 불가 (이 변명은 통하지 않는다)

| 원칙 | 변명 | 현실 |
|------|------|------|
| TDD | "너무 단순해서 테스트 불필요" | 단순한 코드도 깨진다 |
| TDD | "나중에 테스트 추가하겠다" | 나중에 쓴 테스트는 happy path만 커버 |
| 불변성 | "성능 때문에 변이 필요" | 프로파일링으로 증명한 후에만 |
| 시크릿 | "테스트 환경이니까" | 커밋된 테스트 시크릿은 영구 노출 |
| 파일 크기 | "충분히 작다" | 400줄 이상이면 분리 검토 |
| 증거 기반 | "이미 잘 동작한다" | 증거 없는 주장은 거짓 |
| 외과적 | "여기 온 김에 정리하자" | 요청된 줄만 변경. 정리는 별도 요청 |
| 하드게이트 | "빠른 수정이라 설계 불필요" | 3개 파일 변경 = 설계 먼저 |
