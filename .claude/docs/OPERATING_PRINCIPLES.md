# Multi-Agent 운영 원칙 (Operating Principles)

> 버전: 1.1.0
> 최종 갱신: 2026-02-03
> 작성자: System Orchestrator (Opus 4.5)

---

## 1. 핵심 철학

### 1.1 점진적 확장 (Start Small, Scale Gradually)
- **최소 실행 단위**로 시작하여 점진적으로 복잡성 확장
- 단일 에이전트로 검증 후 멀티 에이전트로 확장
- 각 단계에서 안정성 확보 후 다음 단계 진행

### 1.2 투명성 (Transparency First)
- 모든 작업, 결정, 상태 변화를 로깅
- 에이전트 간 통신 내역 추적 가능
- 실패 원인 분석을 위한 충분한 컨텍스트 보존

### 1.3 책임 분리 (Clear Separation of Concerns)
- 각 에이전트는 명확히 정의된 도메인만 담당
- 권한 경계 엄격 준수
- 다른 에이전트 영역 침범 금지

### 3.2 팀 설계 회의 (Design Discussion)

PM-Planner가 태스크를 생성하기 전에, 관련 에이전트 전원이 참여하는 설계 회의를 진행한다.
이 회의는 요구사항을 다각도로 분석하고, 잠재적 문제를 사전에 발견하며, 최적의 구현 방향을 합의하기 위함이다.

**회의 트리거 조건:**
- 신규 기능 개발 요청
- 아키텍처 변경이 예상되는 요청
- 복잡도 M 이상으로 예상되는 요청
- 복수 에이전트 협업이 필요한 요청

**회의 진행 순서:**

```
1. Orchestrator: 요청 분석 및 회의 소집
   - 요청 요약
   - 참석 에이전트 지정
   - 논의 목표 설정

2. PM-Planner: 요구사항 정리
   - 사용자 요구사항 분석
   - 기능/비기능 요구사항 분류
   - 우선순위 제안
   - 의문점/모호한 부분 목록화

3. Architect: 설계 방향 제시
   - 아키텍처 선택지 제안 (최소 2개)
   - 각 선택지의 장단점
   - 기존 시스템과의 호환성 분석
   - 기술 부채 영향 평가

4. 전원 토론 (Round Table):
   - BE-Developer: 구현 난이도, 성능 우려, DB 영향도
   - FE-Developer: UI/UX 복잡도, 상태 관리 영향, 접근성
   - Security-Developer: 보안 위협 모델링, 인증/인가 영향, 공격 표면 분석
   - AI-Engineer: AI/ML 요소 식별, 모델 선택, 데이터 요구사항
   - QA-DevOps: 테스트 전략, 배포 영향, 인프라 요구사항
   - Designer: UI/UX 시안 방향, 디자인 제약사항

5. 합의 도출:
   - 채택된 설계 방향 정리
   - 남은 리스크 목록
   - 추가 조사 필요 항목

6. PM-Planner: 회의 결과 기반 태스크 분해
   - 태스크 목록 초안 작성
   - 의존성 관계 정의
   - 담당 에이전트 배정
```

**회의록 형식:**

```
[설계 회의록] — [주제] / [날짜]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

📋 요청 요약: [1~2줄]
👥 참석: [에이전트 목록]

🎯 PM-Planner 요구사항 분석:
- [핵심 요구사항]

🏗️ Architect 설계 제안:
- 안 A: [설명] — 장점/단점
- 안 B: [설명] — 장점/단점
- 권장: [선택 근거]

💬 라운드 테이블:
- BE-Developer: [의견]
- FE-Developer: [의견]
- Security-Developer: [보안 관점]
- AI-Engineer: [의견]
- QA-DevOps: [의견]
- Designer: [의견]

✅ 합의 사항:
- 채택: [결정]
- 리스크: [목록]
- 추가 조사: [있으면]

📝 → decisions-log.md에 기록
📋 → PM-Planner가 태스크 분해 착수
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**간이 회의 (Quick Sync):**
복잡도 S 이하이거나 단일 에이전트 작업인 경우, 풀 회의 대신 간이 회의를 진행:
- Orchestrator + 담당 에이전트 + Security-Developer(보안 관련 시) 만 참석
- 5분 내 방향 합의
- 간략 메모만 기록

### 3.3 작업 시작 체크리스트

1. **컨텍스트 로딩**
   ```
   □ CLAUDE.md 읽기 (마스터 헌장)
   □ handoff/current.md 확인 (현재 상태)
   □ tasks/TODO.md 검토 (할당된 작업)
   □ context/decisions-log.md 참조 (선례 확인)
   ```

2. **작업 선언**
   ```
   □ 태스크 상태를 IN_PROGRESS로 변경
   □ 시작 시간 기록
   □ 예상 산출물 명시
   ```

3. **의존성 확인**
   ```
   □ 선행 태스크 완료 여부
   □ 필요한 파일/리소스 존재 여부
   □ 다른 에이전트와 충돌 가능성
   ```

### 3.4 작업 완료 체크리스트

1. **품질 검증**
   ```
   □ 코드 린팅 통과
   □ 유닛 테스트 통과
   □ 타입 체크 통과
   □ 통합 테스트 통과 (해당 시)
   ```

2. **문서화**
   ```
   □ 변경 사항 기록
   □ 핸드오프 문서 작성
   □ decisions-log.md 업데이트 (중요 결정 시)
   ```

3. **인수인계**
   ```
   □ 태스크 상태를 REVIEW로 변경
   □ handoff/current.md 갱신
   □ 다음 에이전트에게 필요한 컨텍스트 전달
   ```

## 4. 통신 프로토콜

### 4.1 에이전트 간 메시지 형식

```json
{
  "from": "BE-Developer",
  "to": "QA-DevOps",
  "type": "HANDOFF | REQUEST | NOTIFY | ESCALATE",
  "priority": "P0 | P1 | P2",
  "task_id": "TASK-042",
  "content": {
    "message": "API 엔드포인트 구현 완료. 테스트 요청.",
    "files_changed": ["api/users.py", "models/user.py"],
    "test_hints": ["POST /api/users 생성 테스트 필요"]
  },
  "timestamp": "2026-02-03T14:30:00Z"
}
```

### 4.2 에스컬레이션 규칙

| 상황 | 에스컬레이션 대상 | 응답 시간 |
|------|------------------|-----------|
| 기술적 차단 | Architect | 즉시 |
| 요구사항 모호 | PM-Planner | 즉시 |
| 에이전트 간 충돌 | Orchestrator | 즉시 |
| 보안 우려 (Medium 이하) | Security-Developer | 즉시 |
| 보안 우려 (High 이상) | Security-Developer → Human Lead | 즉시 |
| 일반 질문 | 담당 에이전트 | 다음 턴 |

### 4.3 충돌 해결 우선순위

1. Human Lead 지시
2. CLAUDE.md 규칙
3. 아키텍처 결정 로그
4. 선임 에이전트 판단 (Opus > Sonnet > Haiku)
5. 타임스탬프 기준 선착순

## 5. 로깅 및 모니터링

### 5.1 필수 로그 항목

```python
# 모든 에이전트가 기록해야 하는 정보
{
    "timestamp": "ISO 8601 형식",
    "agent_id": "에이전트 식별자",
    "action": "수행한 작업",
    "target": "대상 파일/리소스",
    "status": "SUCCESS | FAILURE | PENDING",
    "duration_ms": "소요 시간",
    "context": {
        "task_id": "관련 태스크",
        "session_id": "세션 ID",
        "parent_action": "상위 작업 ID"
    },
    "details": "추가 정보"
}
```

### 5.2 로그 레벨 정의

| 레벨 | 용도 | 보존 기간 |
|------|------|-----------|
| ERROR | 실패, 예외 | 90일 |
| WARN | 잠재적 문제 | 30일 |
| INFO | 주요 상태 변화 | 14일 |
| DEBUG | 상세 디버깅 | 7일 |
| TRACE | 세부 추적 | 1일 |

### 5.3 모니터링 지표

- **처리량**: 시간당 완료 태스크 수
- **실패율**: 태스크 실패 비율
- **대기 시간**: 태스크 할당-시작 간 지연
- **핸드오프 효율**: 에이전트 간 전환 횟수
- **재작업률**: REJECTED 후 재시도 비율

## 6. 오류 처리 및 복구

### 6.1 오류 분류

| 유형 | 설명 | 복구 전략 |
|------|------|-----------|
| 일시적 오류 | 네트워크, 타임아웃 | 자동 재시도 (3회) |
| 논리 오류 | 잘못된 입력/로직 | 태스크 반려, 재할당 |
| 시스템 오류 | 의존성 장애 | 에스컬레이션 |
| 인적 오류 | 잘못된 지시 | 확인 요청 |

### 6.2 자동 복구 절차

```
1. 오류 감지 및 분류
2. 로그 기록 (상세 컨텍스트 포함)
3. 재시도 가능 여부 판단
   ├─ 가능 → 재시도 (지수 백오프)
   └─ 불가능 → 에스컬레이션
4. 상태 롤백 (필요 시)
5. 알림 발송
```

### 6.3 안전 장치

- **데드락 방지**: 순환 의존성 검사
- **무한 루프 방지**: 최대 재시도 횟수 제한
- **리소스 보호**: 동시 접근 락
- **상태 보존**: 체크포인트 저장

## 7. 보안 및 접근 제어

### 7.1 민감 정보 처리

Security-Developer가 보안 정책 수립 및 집행을 담당한다.

```
금지 사항:
- .env 파일 커밋
- 하드코딩된 API 키/비밀번호
- 로그에 민감 정보 기록
- 외부 전송 시 암호화 미적용

필수 사항:
- 환경 변수 사용
- 시크릿 매니저 활용
- 접근 로그 기록
- 정기적 키 로테이션
```

**OWASP Top 10 상시 점검:**

| 코드 | 항목 | 자가 점검 항목 |
|------|------|---------------|
| A01 | Broken Access Control | 권한 검사 누락 없는지 확인 |
| A02 | Cryptographic Failures | 민감 데이터 암호화 여부 |
| A03 | Injection | 파라미터 바인딩 사용 여부 |
| A04 | Insecure Design | 위협 모델링 수행 여부 |
| A05 | Security Misconfiguration | 기본값/디버그 모드 비활성화 |
| A06 | Vulnerable Components | 의존성 취약점 스캔 |
| A07 | Auth Failures | 인증 로직 검증 |
| A08 | Software/Data Integrity | 서명 검증, CI/CD 보안 |
| A09 | Security Logging Failures | 보안 이벤트 로깅 |
| A10 | SSRF | 외부 URL 검증 |

**의존성 보안 관리 5단계 절차:**

```
1. 신규 의존성 추가 전
   - Security-Developer에게 사전 스캔 요청
   - 라이선스 호환성 확인
   - 유지보수 상태 확인 (마지막 업데이트, 이슈 대응)

2. 정기 스캔 (주 1회)
   - pip-audit / npm audit 실행
   - 결과 문서화

3. 취약점 발견 시
   - 심각도 분류 (Critical/High/Medium/Low)
   - Critical/High: 즉시 업데이트 또는 대체
   - Medium: 1주 내 조치
   - Low: 백로그 등록

4. 업데이트 적용
   - 테스트 환경에서 호환성 검증
   - 변경 로그 확인
   - 점진적 롤아웃

5. 문서화
   - 변경 사항 decisions-log.md 기록
   - 취약점 대응 이력 보관
```

### 7.2 파일 접근 권한

| 경로 | 읽기 | 쓰기 | 설명 |
|------|:----:|:----:|------|
| `CLAUDE.md` | ALL | Human | 마스터 헌장 |
| `agents/*.md` | ALL | Orchestrator | 에이전트 정의 |
| `tasks/*` | ALL | 담당 에이전트 | 태스크 관리 |
| `handoff/*` | ALL | 작업 완료자 | 인수인계 |
| `.env*` | 실행 시 | Human | 환경 설정 |

## 8. 성능 최적화

### 8.1 병렬 처리 가이드라인

- 독립적인 태스크는 동시 실행
- 의존성 있는 태스크는 순차 실행
- 리소스 경합 시 우선순위 기반 스케줄링

### 8.2 컨텍스트 관리

```
효율적인 컨텍스트 사용:
- 필수 정보만 전달
- 중복 로딩 방지 (캐싱)
- 대용량 파일 분할 처리
- 요약본 우선 사용
```

### 8.3 모델 선택 기준

| 작업 유형 | 권장 모델 | 이유 |
|----------|----------|------|
| 복잡한 설계 | Opus 4.5 | 높은 추론 능력 |
| 일반 구현 | Sonnet 4.5 | 균형잡힌 성능 |
| 반복 작업 | Haiku 4.5 | 빠른 응답 |
| 비주얼 작업 | Gemini 2.0 | 멀티모달 |

### 8.5 자율 QA 사이클 (Self-QA Loop)

모든 코드 산출물 납품 전 5단계 검증을 수행한다:

```
[1] 정적 분석
    - Python: ruff, mypy, black
    - TypeScript: ESLint, tsc, prettier
    - 기준: 에러 0건

[2] 테스트
    - 단위 테스트 실행
    - 커버리지 확인 (80% 이상)
    - 기준: 전 테스트 PASS

[3] 보안 스캔
    - OWASP Top 10 자가 점검
    - 의존성 스캔 (pip-audit / npm audit)
    - 시크릿 검사 (하드코딩 여부)
    - 기준: Critical/High 0건

[4] 성능 확인
    - N+1 쿼리 검사
    - 불필요한 루프 검사
    - 페이지네이션 적용 확인
    - 기준: 명백한 성능 이슈 없음

[5] 종합 판정
    - FAIL 0건 → 납품 가능
    - FAIL 1건 이상 → 수정 후 재검증
```

**교차 검증 매트릭스 (최소 2인 리뷰):**

| 작성자 | 리뷰어 1 | 리뷰어 2 |
|--------|----------|----------|
| BE-Developer | Architect | Security-Developer |
| FE-Developer | BE-Developer | Security-Developer |
| AI-Engineer | Architect | BE-Developer |
| Security-Developer | Architect | QA-DevOps |
| QA-DevOps | Security-Developer | BE-Developer |

## 9. 지속적 개선

### 9.1 KPT 회고 프로세스

프로젝트/스프린트 종료 시 KPT 형식으로 회고를 진행한다:

```
🟢 Keep (유지할 것)
- 잘 작동한 프로세스
- 효과적이었던 도구/방법
- 팀 협업에서 좋았던 점

🔴 Problem (문제점)
- 비효율적이었던 부분
- 예상치 못한 장애물
- 개선이 필요한 프로세스

🔵 Try (시도할 것)
- Problem 해결을 위한 구체적 액션
- 새로 도입할 도구/방법
- 실험해볼 아이디어

📝 기술 부채 (Tech Debt)
- 발견된 기술 부채 목록
- 우선순위 및 예상 영향
- context/tech-debt.md에 기록

🎯 액션 아이템 (Action Items)
- [ ] 구체적인 다음 단계
- [ ] 담당자 지정
- [ ] 기한 설정
```

### 9.2 메트릭 기반 개선

```
추적 지표:
- 태스크 완료율
- 평균 사이클 타임
- 코드 품질 점수
- 테스트 커버리지
- 재작업 비율
```

### 9.3 피드백 루프

```
Human Lead
    ↓ 피드백
Orchestrator
    ↓ 조정
에이전트들
    ↓ 실행 결과
Metrics
    ↓ 분석
개선안 도출 → Human Lead
```
